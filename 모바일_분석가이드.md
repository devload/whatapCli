# 모바일 앱 분석 가이드

이 문서는 모바일 앱 모니터링 데이터를 분석하여 크래시, ANR, 성능 문제를 진단하고 해결하는 방법을 설명합니다.

---

## 1. 이상 징후 감지

### 1.1 모니터링해야 할 핵심 지표

| 지표 | 정상 범위 | 주의 | 심각 |
|------|----------|------|------|
| 크래시율 (Crash-free Rate) | > 99% | 97-99% | < 97% |
| ANR 발생률 | < 0.1% | 0.1-0.5% | > 0.5% |
| 앱 시작 시간 (Cold Start) | < 2초 | 2-5초 | > 5초 |
| 앱 시작 시간 (Warm Start) | < 1초 | 1-2초 | > 2초 |
| 메모리 사용량 | < 70% | 70-85% | > 85% |
| 예외 발생률 | < 1% | 1-3% | > 3% |

### 1.2 이상 징후 패턴

**패턴 A: 특정 기기에서만 크래시**
```
증상: Samsung Galaxy S23에서만 크래시 집중
의심: 기기별 호환성, 해상도/메모리 이슈
```

**패턴 B: 특정 OS 버전에서 ANR**
```
증상: Android 14에서 ANR 다수 발생
의심: OS 변경사항 대응 미흡, 권한/백그라운드 정책
```

**패턴 C: 특정 화면 진입 시 크래시**
```
증상: 결제 화면 진입 시 100% 크래시
의심: 해당 기능 버그, 의존성 문제
```

**패턴 D: 앱 버전 업 후 크래시율 급증**
```
증상: 2.0.0 배포 후 크래시율 5% → 15%
의심: 새 버전 회귀 버그
```

### 2.2 ANR (Application Not Responding) → 원인 찾기

```
ANR 발생
    │
    ▼

    │
┌───┴────┐

│        │
▼        ▼

작업?   부족?
│
├─ 디스크 읽기/쓰기

```

**분석 명령어:**

```bash

LIMIT 20"

device              osVersion    anr_count
Samsung Galaxy A10  Android 12   15
Samsung Galaxy A20  Android 13   12
Xiaomi Redmi 9      Android 11   8

# 해석: 저가형 기기에서 집중
# → 메모리/성능 문제 의심
```

```bash
# 2. ANR 발생 시점 세션 정보
$ whatap mxql --pcode 12345 "CATEGORY mobile_anr

SELECT [activity, threadState, anrReason, @timestamp]

activity          threadState    anrReason
```

## 3. 실제 트러블슈팅 케이스

### 케이스 1: 신규 기능 배포 후 크래시율 급증

**증상:**
- 3.0.0 배포 후 크래시율 0.5% → 8%
- iOS 사용자만 영향

**분석 과정:**

```bash
# Step 1: 크래시 유형 확인
$ whatap mxql --pcode 12345 "CATEGORY mobile_crash
TAGLOAD
SELECT [crashType, count, appVersion, os]
FILTER { appVersion == '3.0.0' }
LIMIT 10"

crashType             count   appVersion  os
FoundationException   1,234   3.0.0       iOS
NullPointerException   56      3.0.0       Android

# Step 2: iOS 크래시 상세
$ whatap mxql --pcode 12345 "CATEGORY mobile_crash
TAGLOAD
SELECT [crashMessage, osVersion, device]
FILTER { crashType == 'FoundationException' AND os == 'iOS' }
LIMIT 5"

crashMessage                         osVersion  device
Unexpectedly found nil unwrap...     iOS 17.2   iPhone 15
Unexpectedly found nil unwrap...     iOS 17.1   iPhone 14

# Step 3: 스택트레이스 확인
$ whatap mxql --pcode 12345 "CATEGORY mobile_crash
TAGLOAD
SELECT [crashMessage, stacktrace]
FILTER { crashType == 'FoundationException' }
LIMIT 1"

crashMessage: Unexpectedly found nil while unwrapping Optional
stacktrace:
  at CartViewModel.fetchProducts()
  at CartViewModel.$fetchProducts()
  at closure #1 in CartViewModel.load()
```

**원인:** 3.0.0에서 새로 추가된 장바구니 기능에서 옵셔널 언래핑 실패

**해결:**
```swift
// Before (크래시)
let product = products.first!.name

// After
guard let product = products.first else { return }
let name = product.name
```

### 케이스 2: 특정 Android 기기에서만 ANR 반복

**증상:**
- Samsung 저가형 기기에서 ANR 반복
- 결제 화면에서 발생

**분석 과정:**

```bash
# Step 1: ANR 발생 기기 패턴
$ whatap mxql --pcode 12345 "CATEGORY mobile_anr
TAGLOAD
SELECT [device, activity, anrReason]
LIMIT 20"

device              activity        anrReason
Samsung Galaxy A10  PaymentActivity disk I/O
Samsung Galaxy A20  PaymentActivity disk I/O
Samsung Galaxy A10  PaymentActivity network wait

# Step 2: 세션 정보 확인
$ whatap mxql --pcode 12345 "CATEGORY mobile_device_session
TAGLOAD
SELECT [memoryUsage, anr_count, device]
FILTER { device like 'Samsung Galaxy A' }
LIMIT 10"

memoryUsage  anr_count  device
85%          5          Samsung Galaxy A10
78%          3          Samsung Galaxy A20

# Step 3: 문제 코드 위치 파악
$ whatap mxql --pcode 12345 "CATEGORY mobile_anr
TAGLOAD
SELECT [stacktrace, anrReason]
FILTER { activity == 'PaymentActivity' }
LIMIT 1"

stacktrace:
  at PaymentActivity.saveTransaction()
  at SharedPreferencesImpl.writeToFile()
  at SharedPreferencesImpl.awaitCommit()

# 해석: 결제 정보를 SharedPreferences에 동기식 저장
# 저사양 기기에서 디스크 I/O로 메인스레드 블로킹
```

**원인:** 메인스레드에서 SharedPreferences commit() 호출

**해결:**
```kotlin
// Before (ANR)
sharedPreferences.edit().putString("tx", data).commit()

// After
// 1. apply() 사용 (비동기)
sharedPreferences.edit().putString("tx", data).apply()

// 2. 또는 백그라운드 스레드
withContext(Dispatchers.IO) {
    sharedPreferences.edit().putString("tx", data).commit()
}
```

### 케이스 3: 앱 시작 시간 5초 → 이탈률 증가

**증상:**
- 앱 시작 시간 평균 5초
- 첫 화면 이탈률 40%

**분석 과정:**

```bash
# Step 1: 시작 시간 분해
$ whatap mxql --pcode 12345 "CATEGORY mobile_app_start_detail
TAGLOAD
SELECT [phase, duration]
LIMIT 20"

phase                 duration
Application.onCreate  2.3s   ← 문제
MainActivity.onCreate 1.8s
NetworkSync           1.2s
UIRendering           0.5s

# Step 2: Application 초기화 내역
$ whatap mxql --pcode 12345 "CATEGORY mobile_init_trace
TAGLOAD
SELECT [library, duration]
FILTER { phase == 'Application.onCreate' }
LIMIT 10"

library               duration
AnalyticsSDK          0.8s
CrashReporter         0.3s
PushNotification      0.5s
ImageLoader           0.4s
DatabaseInit          0.3s

# 해석: AnalyticsSDK 초기화만 0.8초
# → 실제로는 SDK들이 순차 초기화되어 총 2.3초
```

**원인:** Application.onCreate에서 다수 SDK 동기식 초기화

**해결:**
```kotlin
// Before (순차 초기화)
override fun onCreate() {
    AnalyticsSDK.init(this)
    CrashReporter.init(this)
    PushSDK.init(this)
    ImageLoader.init(this)
}

// After (지연/병렬 초기화)
override fun onCreate() {
    // 필수만 동기
    CrashReporter.init(this)

    // 나머지는 비동기
    CoroutineScope(Dispatchers.IO).launch {
        AnalyticsSDK.init(this@MyApp)
        PushSDK.init(this@MyApp)
        ImageLoader.init(this@MyApp)
    }
}
```

## 4. 핵심 분석 체크리스트

### 크래시 급증 시

- [ ] 특정 앱 버전? → 회귀 버그 의심
- [ ] 특정 OS 버전? → OS 호환성 문제
- [ ] 특정 기기? → 기기별 대응 필요
- [ ] 특정 화면? → 해당 기능 집중 분석
- [ ] 네트워크 에러 동반? → API 연동 문제

### ANR 발생 시

- [ ] 저사양 기기 집중? → 성능 최적화
- [ ] 메인스레드 I/O? → 백그라운드 처리
- [ ] 락 대기? → 동기화 문제
- [ ] 네트워크 대기? → 타임아웃/비동기
- [ ] 메모리 부족? → 메모리 누수 확인

### 앱 시작 느릴 때

- [ ] Cold Start > 3초? → 초기화 최적화
- [ ] SDK 초기화 과다? → 지연 초기화
- [ ] 네트워크 요청 동기? → 비동기 전환
- [ ] 리소스 로딩? → 지연 로딩
- [ ] DB 마이그레이션? → 백그라운드 처리
